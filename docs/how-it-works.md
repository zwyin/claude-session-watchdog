# Claude Code Session Watchdog — 系统工作原理

## 一句话概括

每 15 秒采样所有 tmux Claude Code 会话，通过三路联合信号检测卡住的会话，10 分钟告警、15 分钟自动干预恢复；对空闲会话用关键字 + LLM 做语义分类后通知；定时发送早晚报汇总。

## 1. 会话发现

**目标**：找到当前所有运行 Claude Code 的 tmux 会话。

```
ps -o ppid,command          tmux list-sessions       匹配结果
获取所有进程 ──────────→ 筛选 claude/claude-yes ──→ 获取 tmux pane PID ──→ 会话列表
```

**流程**：

1. `ps -o ppid,command` 获取所有进程，筛出含 `claude-yes`、`agent-yes` 或 `/claude` 的进程
2. 提取这些进程的父进程 PID (PPID)
3. 遍历 `tmux list-sessions`，获取每个 tmux 会话的 pane PID
4. 两边 PID 匹配的会话即为 Claude Code 会话

**输入**：系统进程表 + tmux 会话列表
**输出**：Claude Code 会话名列表（如 `gps`、`html`、`kwcode`）

## 2. 状态路由 — 每个会话的核心判断

对每个发现的会话，首先判断它当前处于什么状态：

```
                     ┌──────────────┐
                     │ 采样会话输出  │
                     └──────┬───────┘
                            │
                            ▼
                     ┌──────────────┐
                ┌───→│ 是否空闲？    │
                │    │ is_idle_prompt│
                │    └──┬───────┬───┘
                │       │ 是    │ 否
                │       ▼       ▼
                │  ┌─────────┐ ┌──────────────┐
                │  │ 空闲分支 │ │ 活跃分支      │
                │  │(Section 3)│ │(Section 4)   │
                │  └─────────┘ └──────────────┘
                │
                │  idle → active: 清除空闲状态
                │  stuck → idle:  发送恢复通知，清状态
                │  active → active: 更新 hash/token，重置卡住标记
                └── 下一个会话
```

**空闲判定条件**（满足任一即为空闲）：

| 条件 | 含义 |
|------|------|
| 最后一行是纯 `❯` | Claude Code 空闲提示符 |
| 含 `accept edits on` | 等待权限确认 |
| 含 `[超时]` | API 超时等待 |
| 最后一行是 `目录:` | 状态栏空闲表现 |

## 3. 空闲分支 — 分类通知

当会话被判定为空闲后，等待 10 分钟（`IDLE_CLASSIFY_THRESHOLD`）触发分类：

```
会话空闲 ──→ 记录 idle_since 时间戳
     │
     │  空闲满 10 分钟？ 且未通知过？
     │         │
     │         ▼
     │  ┌─────────────────────────────────────┐
     │  │        classify_idle.py 分类        │
     │  │                                     │
     │  │  ① 捕获末尾 50 行有效输出（过滤噪音）  │
     │  │  ② 关键字匹配（快速路径）             │
     │  │  ③ LLM 语义分析（默认模式，全部走 LLM）│
     │  │  ④ 关键字兜底（LLM 超时时降级）       │
     │  └──────────────┬──────────────────────┘
     │                 │
     │                 ▼
     │  ┌──────────────────────────────────────┐
     │  │ 分类结果 → 事件名 → 通知模板           │
     │  │                                      │
     │  │ decision_needed → idle_decision (黄色)│
     │  │ task_complete   → idle_complete (绿色)│
     │  │ ambiguous       → idle_decision (黄色)│
     │  │ idle_unknown    → idle_unknown  (蓝色)│
     │  └──────────────────────────────────────┘
     │
     └──→ 设置 idle_notified=1（防止重复通知）
```

### 3.1 关键字匹配规则

**决策类关键字**（匹配 → `decision_needed`）：
- 中文：`你觉得`、`选.*方案`、`确认一下`、`需要你决定`
- 英文：`should I`、`which approach`、`what do you think`

**完成类关键字**（匹配 → `task_complete`）：
- 中文：`已经完成`、`代码已提交`、`请查看`
- 英文：`I've completed`、`ready for review`

**排除规则**（仅检查末尾 5 行）：
- `accept edits on` — claude-yes 自动处理
- 独立的 `Yes`/`No` 行 — 权限确认

### 3.2 LLM 语义分析

**输入**：末尾 50 行有效输出（已过滤状态栏、分隔线、空行等噪音）
**处理**：发送给 LLM（主用端点 + 备用端点），附带 Claude Code UI 符号说明
**输出**：`{category, confidence, reasoning, summary}`

**LLM 提示词中的 UI 符号教育**：

| 符号 | 含义 | 分类影响 |
|------|------|---------|
| `✳ Running task…` | 正在执行 | 绝对不是完成 |
| `✻ Worked for Xm Xs` | 阶段耗时 | 不能当判断依据 |
| `✔ item` | 子任务完成 | 不等于整体完成 |
| `◼ item` | 待处理子任务 | 有 ◼ = 未完成 |

## 4. 活跃分支 — 三路联合卡住检测

当会话不是空闲状态时，运行三路信号联合检测：

```
┌─────────────────────────────────────────────────────────────┐
│                    三路信号采集                                │
│                                                             │
│  信号 A: 屏幕哈希            信号 B: JSONL 日志              │
│  ┌───────────────────┐      ┌───────────────────┐          │
│  │ capture-pane 50行  │      │ jsonl_age.py      │          │
│  │ 去除计时器噪音      │      │ 读会话 JSONL 文件  │          │
│  │ 计算 MD5 哈希      │      │ 取最后记录时间戳    │          │
│  │ 与上次比较         │      │ 计算距今秒数       │          │
│  └────────┬──────────┘      └────────┬──────────┘          │
│           │                          │                      │
│           ▼                          ▼                      │
│      hash 不变？              JSONL 停滞 ≥ 10min？           │
│                                                             │
│                    信号 C: 输出 Token                         │
│                    ┌───────────────────┐                     │
│                    │ 状态栏提取 "输出:" │                     │
│                    │ 如 "228.9k"       │                     │
│                    │ 与上次比较         │                     │
│                    └────────┬──────────┘                     │
│                             │                                │
│                             ▼                                │
│                       Token 不变？                           │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    联合判定逻辑                                │
│                                                             │
│  路径 A (经典检测):  hash 不变                      → 卡住    │
│  路径 B (深度检测):  hash 在变 + JSONL 停滞 + Token 不变     │
│                                          → 深度卡住          │
│                                                             │
│  任一路径命中 → 记录 unchanged_since，计算持续时长             │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    阶梯式响应                                  │
│                                                             │
│  持续 0-10 分钟    → 仅记录状态，不动作                       │
│  持续 ≥ 10 分钟    → 发送告警通知（stuck 模板，红色卡片）       │
│  持续 ≥ 15 分钟    → 自动干预（发送 Ctrl-C + 继续指令）        │
│                      + 冷却期 10 分钟防止频繁干预              │
│                                                             │
│  恢复（hash 变化 或 转为空闲）→ 发送恢复通知（绿色卡片）        │
└─────────────────────────────────────────────────────────────┘
```

### 4.1 信号 A 的噪音过滤

计时器（`2m 15s`）和时间戳（`14:30`）会导致屏幕内容持续变化，产生误判。
哈希计算前先用 sed 归一化：

```
"5m 30s"  → "TIMER"
"14:30pm" → "TIME"
```

### 4.2 信号 B 的 JSONL 定位

`jsonl_age.py` 需要找到会话对应的 JSONL 日志文件：

```
tmux session name
    │
    ▼
tmux display-message → 获取会话工作目录
    │
    ▼
工作目录 → 映射到 ~/.claude/projects/<path>/ 目录
    │
    ▼
读取目录下最新的 .jsonl 文件 → 取最后一条记录的 timestamp
    │
    ▼
计算距今秒数 → 判断是否停滞
```

### 4.3 自动干预流程

```
intervene() 函数执行序列：

  tmux send-keys Escape     ← 清除当前输入
        │
        ▼ (等 0.3s)
  tmux send-keys Ctrl-C     ← 中断挂起的 API 请求
        │
        ▼ (等 3s)
  tmux send-keys "继续刚才的任务。如果当前方案卡住了，把任务拆小再执行"
        │
        ▼ (等 0.5s)
  tmux send-keys Enter      ← 提交继续指令
        │
        ▼
  记录 auto_interrupt 事件 + 发送干预通知（橙色卡片）
```

## 5. 通知系统

### 5.1 通知管道

```
watchdog.sh                    notify.py                    飞书 API
┌──────────┐    key=value    ┌────────────┐   JSON + HMAC   ┌────────┐
│ 构建参数  │ ──────────────→ │ 加载模板    │ ─────────────→ │ 渲染卡片│
│ 调用模板  │                │ 替换占位符  │                │ 展示通知│
│ 引擎函数  │                │ 清理特殊字符│                └────────┘
└──────────┘                │ HMAC 签名  │
                            │ POST 请求  │
                            └────────────┘
```

### 5.2 签名流程

```
timestamp = 当前 Unix 时间戳
string_to_sign = "{timestamp}\n{secret}"
sign = Base64(HMAC-SHA256(string_to_sign))
url = webhook_url?timestamp={timestamp}&sign={sign}
```

### 5.3 模板类型总览

| 模板 | 颜色 | 触发条件 | 关键内容 |
|------|------|---------|---------|
| `stuck` | 🔴 红色 | 10 分钟无输出 | 会话名、时长、运行状态、末尾输出 |
| `intervene` | 🟠 橙色 | 15 分钟自动干预 | 会话名、时长、动作、运行状态 |
| `recovered` | 🟢 绿色 | 卡住后恢复 | 会话名、中断时长 |
| `start` | 🔵 蓝色 | 守护进程启动 | 版本号、监控会话数、配置摘要 |
| `idle_decision` | 🟡 黄色 | 空闲 → 需要决策 | 置信度、判断依据、上下文摘要 |
| `idle_complete` | 🟢 绿色 | 空闲 → 任务完成 | 置信度、判断依据、上下文摘要 |
| `idle_unknown` | 🔵 蓝色 | 空闲 → 原因不明 | 置信度、判断依据 |
| `morning_report` | 🟣 紫色 | 每天 08:00 | 夜间事件汇总、会话明细 |
| `evening_report` | 🟣 紫色 | 每天 22:00 | 白天事件汇总、会话明细 |

## 7. 事件日志

所有检测事件追加写入 `~/.claude/session-events.jsonl`，每行一条 JSON：

```json
{
  "timestamp": "2026-05-13T12:00:00Z",   // UTC 时间
  "event": "stuck",                       // 事件类型
  "session": "gps",                       // tmux 会话名
  "project": "gps",                       // 项目名（= 会话名）
  "duration_minutes": 12,                 // 持续时长
  "model": "GLM-5.1",                     // 运行模型
  "phase": "unknown",                     // 阶段（保留字段）
  "intervention": "none",                 // 干预方式
  "recovered": false,                     // 是否已恢复
  "notes": "stuck: hash unchanged...",    // 事件描述
  "context": "..."                        // 上下文（仅部分事件）
}
```

**事件类型**：

| 事件类型 | 含义 |
|---------|------|
| `stuck` | 检测到卡住 |
| `auto_interrupt` | 自动干预 |
| `recovered` | 恢复正常 |
| `idle_decision` | 空闲 → 需要决策 |
| `idle_task_complete` | 空闲 → 任务完成 |
| `idle_unknown` | 空闲 → 原因不明 |

## 8. 状态管理

每个会话的状态存储在 `~/.claude/watchdog-state/<session>.<key>` 文件中，调试时直接 `cat`/`rm` 即可。

### 状态文件列表

| 文件 | 含义 | 写入时机 |
|------|------|---------|
| `<session>.hash` | 上次屏幕哈希 | 每次采样更新 |
| `<session>.unchanged_since` | 卡住起始时间戳 | 首次检测到卡住时写入 |
| `<session>.stuck_notified` | 是否已发送卡住通知（值=1） | 发送通知时写入 |
| `<session>.last_intervene` | 上次干预时间戳 | 干预时写入 |
| `<session>.output_tokens` | 上次输出 token 数 | 每次采样更新 |
| `<session>.idle_since` | 空闲起始时间戳 | 进入空闲时写入 |
| `<session>.idle_notified` | 是否已发送空闲通知（值=1） | 发送通知时写入 |
| `_global.last_morning_date` | 上次早报日期 | 发送早报时写入 |
| `_global.last_evening_date` | 上次晚报日期 | 发送晚报时写入 |

### 完整状态机

```
                        ┌──────────┐
                        │  ACTIVE  │ ← 正常运行中
                        │ (非空闲)  │
                        └─────┬────┘
                              │
              ┌───────────────┼───────────────┐
              │               │               │
         hash/JSONL/     is_idle_prompt  hash 变化
         token 停滞       匹配成功         (之前卡住)
              │               │               │
              ▼               ▼               ▼
        ┌──────────┐   ┌──────────┐   ┌──────────┐
        │  STUCK   │   │   IDLE   │   │ RECOVERED │
        │ (卡住中)  │   │  (空闲)  │   │ (已恢复)  │
        └────┬─────┘   └────┬─────┘   └──────────┘
             │               │
        ┌────┴────┐    ┌────┴─────┐
        │         │    │          │
   10min 无效  15min 无效  10min空闲  用户操作
   → 发送告警  → 自动干预  → LLM分类   → 回到 ACTIVE
        │         │    → 发送分类通知
        ▼         ▼         │
   NOTIFIED  INTERVENED   NOTIFIED
                             │
                        用户操作/超时
                             │
                             ▼
                        回到 ACTIVE
```

## 10. 时间参数全景

## 11. 部署模式

```
方式 A: launchd（推荐，macOS 开机自启）
  com.claude.watchdog.plist
    → RunAtLoad=true, KeepAlive=true
    → watchdog.sh daemon（前台循环）
    → stdout/stderr → watchdog.log

方式 B: fork 后台
  ./watchdog.sh start
    → fork 子进程
    → 后台循环 do_check()
    → 写入 PID 文件

方式 C: 单次运行
  ./watchdog.sh run
    → 执行一次 do_check()
    → 退出（适合 cron 调度）
```
