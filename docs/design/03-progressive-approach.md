# 03 — 渐进策略

## 原则

1. **最小干预**：补丁越少越好，每加一层外部工具都是在增加维护成本
2. **等官方修复**：Claude Code 的流式看门狗、tmux 兼容性都在持续改进，官方方案优于社区方案
3. **数据驱动**：先量化问题（监控），再决定是否升级方案
4. **不破坏上下文**：优先 Ctrl-C + 继续的方式，避免杀进程丢上下文

---

## 第 1 层：内置流式看门狗（立即可用，零成本）

### 配置

在 `.zshrc` 或 tmux 启动脚本中添加：

```bash
# 启用事件级流式看门狗（第三方模型必须手动开启）
export CLAUDE_ENABLE_STREAM_WATCHDOG=1

# 超时时间（毫秒），默认最低 300000 = 5 分钟
export CLAUDE_STREAM_IDLE_TIMEOUT_MS=300000
```

### 作用
- 当 API 代理的 SSE 响应流中断 5 分钟无任何数据 → 自动中断请求 → Claude Code 自动重试
- 不影响正常的长思考（thinking），因为 thinking 时仍有字节流
- 每个跑在 tmux 里的 claude 进程都会自动生效

### 预期效果
- 解决"SSE 流断了傻等 30-60 分钟"的问题
- 5 分钟后自动恢复，不需要人工 Ctrl-C

### 评估方式
- 启用后记录一周的卡住事件，对比启用前（参见 04-monitoring-plan.md）

## 第 2 层：外部通知 + 自动干预（已实现）

> **状态：已实现（v2.0.1）**，三路联合检测 + 飞书通知 + 自动干预 + 空闲分类。

### 触发条件
- 5 分钟空闲 → 空闲分类（关键字 + 可选 LLM）+ 分类通知（仅通知，不干预）
- 10 分钟无有效输出 → 卡住告警通知
- 15 分钟无有效输出 → 自动 Ctrl-C + 继续任务

### 检测方式（三路联合）
1. **屏幕 hash**：`tmux capture-pane` 从最近 50 行中取末尾 20 行，去除计时器干扰后比较 MD5
2. **JSONL 日志**：读取会话对应项目目录下最近修改的 `.jsonl` 文件，检查最后一条记录的时间戳
3. **输出 token 数**：从状态行提取 `输出:` 字段，检测数值是否停滞

任一路判定为卡住即触发；深度检测（hash 变但 JSONL 停滞 + token 不变）也会触发。

### 空闲分类（v2.0.1 新增）

会话空闲 5 分钟后，通过 `classify_idle.py` 进行分类：

1. **关键字匹配**（主路径）：正则匹配中英文决策/完成关键词，排除 claude-yes 自动处理的简单权限确认
2. **LLM 语义分析**（可选）：当关键字匹配结果为 `ambiguous`（同时命中决策+完成）时，调用 LLM API 进行语义判断
3. **降级策略**：LLM 超时或不可用时，回退到关键字结果，通知中标注 LLM 不可用

分类结果 → 不同颜色飞书卡片：
- `decision_needed`（黄色）— 等待人工决策，通知中附带上下文摘要
- `task_complete`（绿色）— 任务完成，等待验收
- `idle_unknown`（蓝色）— 无法自动判断原因

空闲会话**只通知不干预**，仅明确卡住的会话触发自动 Ctrl-C。

### 动作
- macOS 原生通知（`osascript -e 'display notification'`）
- 飞书机器人卡片通知（webhook + HMAC 签名，8 种模板）
- 自动干预：`Escape` → `Ctrl-C` → 发送"继续"消息（仅卡住会话）

## 第 3 层：自动续命（已实现）

> **状态：已实现（v2.0.0）**，作为第 2 层的一部分，15 分钟阈值自动触发。

### 关键：不杀进程
- Ctrl-C 只是中断当前 API 请求，上下文和对话历史完整保留
- 跟用户手动操作一模一样
- 冷却期 10 分钟，避免反复干预

## 回退方案

如果第 1 层导致问题（比如正常的长时间思考被误中断）：
- 调大 `CLAUDE_STREAM_IDLE_TIMEOUT_MS`（如 600000 = 10 分钟）
- 或临时关闭 `CLAUDE_ENABLE_STREAM_WATCHDOG=0`
